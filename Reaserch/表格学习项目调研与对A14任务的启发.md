# 表格学习项目调研与对 A14 任务的启发

## 1. 调研目标

这份调研文档聚焦于我们前面讨论过的一批代表性表格学习项目/论文，回答 3 个问题：

1. 这些项目分别在解决什么问题。
2. 它们对当前 A14 项目是否有直接帮助。
3. 如果要继续改造当前项目，应该优先借鉴哪些思路。

这里的 A14 任务，指当前仓库里的这条主线：

- 输入是学生-学期级别的多源表格数据。
- 目标一是预测 `risk_label_next_term`，也就是“下学期风险”。
- 目标二是学习学生画像与行为模式，并做聚类、异常点识别和干预分层。
- 当前主流程是先做自监督表征学习，再做聚类和风险预测。

从当前本地代码看，项目目前的特征大致是 `5` 个类别特征加 `17` 个数值特征，总计 `22` 维；训练样本按学年做时间切分；风险标签带明显的“跨学期预测”属性。这决定了我们不能只把它当成普通表格分类问题看，而要把它看成“异构表格 + 时间演化 + 稀有风险事件 + 学生画像发现”的组合任务。

---

## 2. 一页结论

### 最核心的判断

- 这些项目里，**没有任何一个可以原样直接替代你的任务设计**。
- 但它们里面有 4 条思路对你特别有价值：
  - `TabularS3L`：适合继续作为自监督表征学习框架。
  - `T2G-Former`：适合解决“异构特征之间关系不均等”的问题。
  - `TabR`：适合解决“相似学生可为当前学生提供参考”的问题。
  - `TabM`：适合做更强、更实用的监督学习基线。

### 对当前项目最重要的结论

- 当前项目的主线思路是合理的，但“**单学期编码**”的味道仍然太重。
- 你的任务真正难的地方，不只是学出单行表格特征，而是学出：
  - 学生跨学期状态如何演化。
  - 哪些特征变化组合会触发风险。
  - 哪些学生与哪些历史学生相似。
- 所以，后续最值得投入的方向不是继续把普通 Transformer 堆得更深，而是：
  - 加入时间建模。
  - 显式利用特征关系。
  - 显式利用相似学生检索。
  - 同时保留强树模型和强 MLP 基线。

### 推荐优先级

| 项目 | 主要价值 | 对当前 A14 的帮助 | 建议优先级 |
|---|---|---|---|
| TabularS3L | 自监督/半监督表征学习框架 | 高 | 高 |
| FT-Transformer / RTDL | 强 tabular Transformer 基线 | 中 | 中 |
| TabTransformer | 类别特征上下文化、抗噪声 | 中 | 中 |
| SAINT | 行列注意力 + 对比预训练 | 中 | 中 |
| Why do tree-based models still outperform DL on tabular data? | 告诉你 tabular 任务的真实难点 | 很高 | 高 |
| Tabular Data: Deep Learning is Not All You Need | 提醒不要忽视 XGBoost/GBDT | 很高 | 高 |
| T2G-Former | 特征关系图 + 异构特征交互 | 很高 | 高 |
| TabR | 检索相似样本、吸收邻居信号 | 很高 | 高 |
| TabM | 强监督学习基线，实用且高效 | 很高 | 高 |

## 3. 任务特点：为什么你的任务比“普通表格分类”更难

在决定借鉴哪个项目之前，先要明确你这个任务的结构。

### 3.1 这是异构表格，不是同质特征向量

你的数据同时包含：

- 人口属性：性别、年级、学院、专业
- 学业表现：均分、GPA、挂科数、补考数、课程数
- 网络行为：上网时长、月均时长、变化量、在线学习占比
- 体测与行为：体测分、BMI 拆解出的身高体重

这些特征的统计性质、语义层级、噪声模式完全不同，因此“所有特征一视同仁”的网络结构往往不够理想。

### 3.2 这是时间问题，不只是分类问题

目标是预测“下学期风险”。这意味着真正有价值的模式往往是：

- 某个学生的成绩是不是在持续下降。
- 网络行为是否突然异常。
- 挂科、补考、上网和体测之间是否出现联合变化。

也就是说，**趋势** 比 **单点数值** 更重要。

### 3.3 这是风险预警问题，异常点很重要

风险学生通常占比较低，异常群体虽然小，但业务价值反而高。对于这类任务：

- 只追求平均分类精度不够。
- 模型对异常点、边界样本、稀有模式的刻画能力很关键。

### 3.4 这是“表示学习 + 分群 + 预测”的复合任务

你不是只想要一个二分类器，还希望：

- 学到表征 embedding
- 做学生群体聚类
- 找到异常群体
- 支持干预策略解释

这意味着模型最好不仅能预测，还要能“组织潜在空间”。

## 4. 逐项调研

## 4.1 TabularS3L

### 它解决什么问题

`TabularS3L` 不是单篇论文，而是一个面向表格数据的**自监督/半监督学习框架库**。它要解决的问题是：表格领域已经有很多自监督和半监督方法，但缺少统一、模块化、可复用的工程框架，导致方法难以比较、难以复现、难以落地。

官方仓库明确写到，它是一个基于 PyTorch Lightning 的统一框架，用来支持表格数据上的 self-supervised 和 semi-supervised learning，并采用“两阶段学习”：先做自监督预训练，再做监督或半监督下游任务。

### 核心思路

- 支持多种方法：DAE、VIME、SubTab、SCARF、SwitchTab 等。
- 允许 embedding 模块和 backbone 模块自由组合。
- 把“先学表示、再下游微调”做成统一接口。

### 对 A14 有什么帮助

帮助是**直接且现实的**。

- 你当前项目已经在用它的思路，尤其是 `DAE`、`feature_tokenizer`、`transformer backbone`。
- 你有大量无标签学生学期样本，很适合先做自监督表征学习。
- 如果未来要系统比较不同自监督目标，TabularS3L 是很好的实验底座。

### 需要注意的地方

- 官方框架中，原始 `DAE / VIME / SubTab / SCARF` 默认 backbone 其实是 `mlp`，而 `transformer` backbone 是和 `feature_tokenizer` 组合使用的。
- 这意味着你当前项目里“DAE + Transformer”的组合是合理的工程混搭，但并不是该库里最原教旨的默认设置。
- 它本身并不替你解决“学生跨学期时序建模”。

### 我的判断

`TabularS3L` 非常适合继续保留，作为**自监督表征学习框架**，但它更像“底盘”，不是最终最贴任务的网络结构。

### 是否推荐

推荐，优先级高。

### 官方来源

- 仓库：<https://github.com/Alcoholrithm/TabularS3L>

## 4.2 FT-Transformer / RTDL

### 它解决什么问题

`Revisiting Deep Learning Models for Tabular Data` 这篇工作主要解决两个问题：

- 表格深度学习论文太多，但实验协议不统一，无法公平比较。
- 需要一个足够强、足够干净、足够可复现的深度学习基线。

这篇论文提出并系统评估了两个强基线：

- ResNet-like MLP
- FT-Transformer

### 核心思路

`FT-Transformer` 的关键不是照搬 NLP Transformer，而是：

- 把每个表格特征映射成 token
- 对 token 做特征间注意力
- 通过 feature tokenizer 适配表格数据

它本质上是一个**面向表格特征交互的 Transformer 基线**。

### 对 A14 有什么帮助

帮助主要在两个方面：

- 它告诉你“普通表格 Transformer 应该长什么样”，适合做强 baseline。
- 你当前项目里用到的 `feature_tokenizer + transformer backbone` 路线，本质上就承接了这条设计脉络。

### 局限

- 它擅长的是“单行表格的特征交互”，不擅长处理时序轨迹。
- 对中等规模 tabular 任务，它不一定比树模型更强。
- 如果不加任务特定约束，它学习到的是通用交互，不一定是你关心的风险路径。

### 我的判断

如果你要证明“我们不是乱用 Transformer，而是走了表格学习里一条成熟路线”，FT-Transformer 非常有价值。  
但如果问“它是不是你这个学生风险任务的最终最优结构”，我会说不是。

### 是否推荐

推荐作为基线和结构参考，优先级中等。

### 官方来源

- 论文：<https://arxiv.org/abs/2106.11959>
- 项目集合：<https://github.com/yandex-research/rtdl>

## 4.3 TabTransformer

### 它解决什么问题

`TabTransformer` 要解决的是：传统表格神经网络里，类别特征通常只是简单 embedding，然后直接拼接，这样无法充分建模“类别特征之间的上下文关系”，也对缺失和噪声不够鲁棒。

### 核心思路

- 用 Transformer 对类别特征 embedding 做上下文化。
- 让每个类别特征的表示，依赖其他类别特征上下文。
- 还支持无监督预训练来学习更强的 contextual embedding。

### 对 A14 有什么帮助

对你任务的帮助是**中等偏上，但不是最关键**。

- 你有 `gender / grade / college / major / semester` 这类类别特征。
- 特别是 `college`、`major`、`semester` 的组合，很可能携带上下文信息。
- 因此，TabTransformer 的思路可以提升对类别结构的建模质量。

### 局限

- 它的主要亮点集中在类别特征，而你的任务里更关键的信号很多来自连续行为特征。
- 它仍然主要是单行表格建模。
- 对“下学期风险”这种跨学期预测问题，它不是最针对的解决方案。

### 我的判断

如果你后续想强化“类别语义上下文”和“抗噪声能力”，TabTransformer 可以借鉴。  
但它对你任务的帮助，大概率不如 T2G-Former、TabR、TabM 直接。

### 是否推荐

推荐阅读，但优先级中等。

### 官方来源

- 论文：<https://arxiv.org/abs/2012.06678>

---

## 4.4 SAINT

### 它解决什么问题

`SAINT` 要解决的是：单纯做列注意力不够，表格任务里还可以利用“样本之间”的关系；同时，在标签较少时，自监督对比预训练也可能帮助提升表现。

### 核心思路

- 列注意力：建模特征与特征之间的关系。
- 行注意力：建模样本与样本之间的关系。
- 对比式预训练：在标签有限时增强表征能力。

### 对 A14 有什么帮助

它对你的启发主要在两点：

- “只做 feature attention 不一定够”，样本之间也可能有结构。
- 自监督预训练和监督任务结合是可行路线。

### 局限

- SAINT 的 row attention 通常是在 batch 维度上做，未必对应真实业务关系。
- 对你这个任务来说，“同批学生互相关注”不一定等于“真实相似学生关系”。
- 如果 batch 组织方式不稳定，row attention 学到的可能是训练噪声而不是稳定结构。

### 我的判断

它很值得参考，但要谨慎套用。  
如果你真的想利用学生之间关系，我更倾向于：

- 用显式图关系
- 或用 TabR 那样的相似样本检索

而不是直接照搬 batch 内 row attention。

### 是否推荐

推荐作为思路来源，优先级中等。

### 官方来源

- 论文：<https://arxiv.org/abs/2106.01342>

## 4.5 Why do tree-based models still outperform deep learning on tabular data?

### 它解决什么问题

这篇论文不是提出新模型，而是在问一个更根本的问题：

> 为什么在很多典型表格任务上，树模型仍然比深度学习强？

这对你非常关键，因为它直接决定你后续该不该继续重压 Transformer。

### 核心结论

论文给出的 3 个 tabular 深度学习难点非常重要：

- 要能抗无信息特征
- 要保留数据的方向性
- 要能学习不规则函数

### 对 A14 有什么帮助

帮助非常大，因为你的数据正好有这些典型特征：

- 多源异构，存在弱信息列和噪声列
- 一些特征方向性非常强，比如“挂科增加”与“成绩下降”
- 风险触发机制可能是不规则的、局部的、非平滑的

这意味着：

- 不要默认深度模型一定优于 GBDT
- 网络结构必须带有强任务归纳偏置
- 强树模型基线必须保留

### 我的判断

这篇不是“可选阅读”，而是你后续技术决策的地基。  
如果不接受这篇论文的提醒，很容易把项目优化方向带偏。

### 是否推荐

强烈推荐，优先级高。

### 官方来源

- 论文：<https://arxiv.org/abs/2207.08815>

## 4.6 Tabular Data: Deep Learning is Not All You Need

### 它解决什么问题

这篇工作做的是更直接的现实校验：

- 很多表格深度学习模型声称优于 XGBoost
- 但这些比较是否公平、是否稳定、是否值得在真实项目中替代树模型

### 核心结论

- 在他们的系统比较中，XGBoost 普遍强于当时那些 tabular DL 模型。
- 而且 XGBoost 调参成本更低。
- 但深度模型和 XGBoost 的 ensemble 往往比单独 XGBoost 更好。

### 对 A14 有什么帮助

它对你的价值是战略性的：

- 你可以放心保留树模型，不会显得“技术不够前沿”。
- 如果最终要拿成绩，`GBDT + 深度表示` 的组合可能比单一深度网络更稳。

### 我的判断

如果你的目标是比赛落地和结果稳定性，这篇论文会让你的路线更务实。  
它不意味着不要做深度学习，而是提醒你：

- 深度学习在这里应该服务于表示、迁移、自监督、模式发现
- 预测性能上必须和树模型正面对比

### 是否推荐

强烈推荐，优先级高。

### 官方来源

- 论文：<https://arxiv.org/abs/2106.03253>

## 4.7 T2G-Former

### 它解决什么问题

`T2G-Former` 的核心问题意识非常适合你：

> 表格特征是异构的，不应该默认所有特征都通过同一种方式充分交互。  
> 更合理的方法是先估计“特征关系图”，再沿着关系图做选择性交互。

### 核心思路

- 学一个 feature relation graph
- 用 relation graph 引导 Transformer 只做更有意义的特征交互
- 用更有结构的 readout 聚合多层信息

### 对 A14 有什么帮助

对你的帮助非常高。

因为你的特征天然分成几类：

- 学业
- 网络行为
- 体测/行为
- 人口统计

这些特征之间的关系显然不是均匀的。  
例如：

- `avg_score` 和 `fail_course_count` 的关系应该很强
- `college` 和某些行为特征可能有中等关系
- `gender` 对某些风险路径可能只是弱调制变量

T2G-Former 正是在解决这类“特征异质性 + 非均匀交互”的问题。

### 我的判断

如果你后续只选一个“比普通 FT-Transformer 更贴你任务”的结构方向，我会优先选 T2G-Former 这一类。  
因为它比普通 Transformer 更符合你的任务本质。

### 是否推荐

强烈推荐，优先级高。

### 官方来源

- 论文：<https://arxiv.org/abs/2211.16887>
- 官方仓库：<https://github.com/jyansir/t2g-former>

## 4.8 TabR

### 它解决什么问题

`TabR` 解决的是一个很有意思的问题：

> 对于当前样本，能不能去训练集中检索相似样本，并利用这些“近邻样本”的特征或标签来帮助预测？

这相当于把“相似学生参考机制”显式纳入模型。

### 核心思路

- 先有一个前馈网络主干
- 中间插入类似 kNN 的检索组件
- 检索最相近的训练样本
- 用注意力式机制抽取邻居提供的有用信号

### 对 A14 有什么帮助

帮助非常高，而且非常贴业务直觉。

因为你这个任务很容易存在下面这种情况：

- 当前学生样本不一定明显高风险
- 但如果看历史上“和他很像”的学生，很多都在下一学期出风险

这时 retrieval 能提供强补充信息。

尤其适合：

- 学生画像类任务
- 相似人群风险迁移
- 样本量中等、模式局部而非全局平滑的任务

### 局限

- 推理时需要维护检索库，部署复杂度高于普通前馈网络。
- 如果特征空间没有被组织好，检索质量会受影响。
- 直接用于跨学期预测时，最好把“只检索历史可见样本”这个时间约束处理好，避免泄漏。

### 我的判断

TabR 是你这个任务非常值得试的方向。  
它甚至比继续单纯升级 Transformer 更像“问题导向改造”。

### 是否推荐

强烈推荐，优先级高。

### 官方来源

- 论文：<https://arxiv.org/abs/2307.14338>
- 官方仓库：<https://github.com/yandex-research/tabular-dl-tabr>

## 4.9 TabM

### 它解决什么问题

`TabM` 的出发点很务实：

> 如果 MLP 本身够简单高效，那么能不能通过“参数高效集成”把它变成更强的 tabular 模型，而不需要走复杂 Transformer 或检索结构？

### 核心思路

- 一个模型同时模拟多个 MLP ensemble member
- 大量共享参数
- 既保留 ensemble 的泛化收益，又减少计算和显存开销

### 对 A14 有什么帮助

帮助非常高，尤其在风险预测主任务上。

原因有 3 个：

- 你的输入维度不高，`22` 维左右，并不天然需要重型 Transformer。
- 你的监督任务本质上仍是表格分类/排序问题。
- 你需要一个足够强、足够稳、训练成本不夸张的深度学习 baseline。

TabM 在这种场景下非常有吸引力。

### 局限

- 它主要是监督学习强基线，并不直接替你做自监督表征学习。
- 它不直接解决时间建模。
- 它也不显式解决聚类空间组织问题。

### 我的判断

如果你接下来想提高“风险预测”这条线的分数，TabM 的优先级应该非常高。  
它可能比“继续优化当前单学期 Transformer 编码器”更快带来收益。

### 是否推荐

强烈推荐，优先级高。

### 官方来源

- 论文：<https://arxiv.org/abs/2410.24210>
- 官方仓库：<https://github.com/yandex-research/tabm>

## 5. 这些项目到底对你有没有帮助

如果站在“当前 A14 项目下一步该怎么做”的角度，我的判断如下。

### 5.1 最有帮助的不是“谁最先进”，而是谁最贴任务

对你最有帮助的方向不是单纯追求最新，而是：

- 能处理异构特征关系
- 能处理跨学期演化
- 能处理相似学生参考
- 能在中等数据规模上稳定训练

### 5.2 实用价值排序

#### 第一梯队：建议优先借鉴

- `TabM`
  - 最适合做更强的监督学习 baseline
  - 很可能直接提升风险预测主任务
- `TabR`
  - 最适合把“相似学生参考”显式纳入模型
  - 与学生画像和风险迁移直觉高度一致
- `T2G-Former`
  - 最适合把异构特征关系结构化建模
  - 比普通 Transformer 更贴你任务
- `TabularS3L`
  - 最适合继续承担自监督表征学习底盘

#### 第二梯队：建议作为结构参考和补充对比

- `FT-Transformer`
  - 很重要的标准基线
- `TabTransformer`
  - 对类别特征上下文化和鲁棒性有帮助
- `SAINT`
  - 对比预训练和 row attention 有启发，但要谨慎落地

#### 第三梯队：不是模型，但必须指导路线

- `Why do tree-based models still outperform deep learning on tabular data?`
- `Tabular Data: Deep Learning is Not All You Need`

这两篇决定你该如何看待 tabular DL，而不是给你现成网络。

## 6. 对当前项目最关键的结构建议

这里给出我认为最值得尝试的一版任务导向结构。

## 6.1 方案建议：两层编码

### 第一层：单学期编码器

作用：

- 把一学期的异构表格特征编码成 `semester embedding`

推荐候选：

- `TabM` 或强 MLP 作为监督基线
- `T2G-Former` 风格关系约束编码器
- `TabularS3L` 继续承担自监督预训练

### 第二层：学生时序编码器

作用：

- 把多个学期的 embedding 按时间顺序编码
- 直接服务于“下学期风险预测”

推荐候选：

- GRU / Temporal Transformer / TCN

这是我认为比继续单纯强化单学期 Transformer 更关键的一步。

## 6.2 方案建议：画像学习与风险预测解耦

当前项目里，embedding 同时承担画像和预测。后面可以更清晰地分成两条：

- `portrait branch`
  - 强调聚类结构、异常点、学生模式分布
- `risk branch`
  - 强调对下学期风险的判别能力

这样可以避免一个目标牵着另一个目标跑偏。

## 6.3 方案建议：加入检索增强

把 `TabR` 的思想转成学生版本：

- 对当前学生当前学期表示
- 检索历史上最相似的学生-学期样本
- 聚合这些相似样本的已知后续风险信息

这对预警场景非常自然。

## 6.4 方案建议：保留树模型与深度模型双线

最终对比分组建议至少包括：

- `CatBoost / LightGBM / XGBoost`
- `TabM`
- `当前 SSL embedding + LR/RF`
- `关系图 Transformer`
- `检索增强模型`

不要让项目走成“只剩 Transformer 一条线”。

## 7. 推荐落地路线

如果按性价比排序，我建议后续迭代按下面顺序推进。

### 阶段 1：把监督学习基线补强

- 加 `CatBoost / LightGBM`
- 加 `TabM`
- 做和当前 `SSL embedding + LR/RF` 的公平对比

目的：

- 先知道风险预测天花板在哪里。

### 阶段 2：把单学期编码器改得更贴任务

- 在当前表征学习里引入特征关系约束
- 重点参考 `T2G-Former`

目的：

- 让 embedding 更适合画像和风险结构。

### 阶段 3：加入跨学期时序建模

- 把学生多个学期连接起来
- 明确预测“下一学期风险”

目的：

- 解决当前最本质的短板。

### 阶段 4：加入相似学生检索

- 重点参考 `TabR`

目的：

- 补足局部模式与邻域参考信号。

## 8. 最终结论

### 哪些项目真的值得你重点借鉴

- 如果你要继续做自监督表征：看 `TabularS3L`
- 如果你要把特征交互做得更合理：看 `T2G-Former`
- 如果你要把“相似学生参考”纳入模型：看 `TabR`
- 如果你要把风险预测结果做强：看 `TabM`
- 如果你要避免路线误判：看两篇树模型对比论文

### 哪些方向不建议单独押注

- 不建议只继续堆“普通 tabular Transformer”
- 不建议只围绕单学期静态样本优化
- 不建议忽视树模型和强 MLP 基线

### 我对当前项目的总体评价

当前项目已经走在一条合理路线上：  
“自监督表征学习 -> 聚类/画像 -> 风险预测” 这条主线是成立的。

但如果要让模型更贴合 A14 任务本身，下一步最需要补的不是“更复杂的普通 Transformer”，而是：

- 时间建模
- 特征关系建模
- 相似学生检索
- 更强监督基线
